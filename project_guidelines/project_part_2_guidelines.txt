Given:


Notation:
	•	Each state has a 2-bit label CS1CS0.
	•	Each transition starts at CS1CS0 , and then points to the matching NS1NS0
	•	Each transition has a multibit label. { T1T0, T1T0}/ Out2Out1Out0

Solve the Selector Logic Problem (TBA points)
Given:

The circuit is a counter circuit with a register and is controlled by the Selector Logic. The Selector Logic has 4 1-bit inputs and 1 4-bit outputs. The Selector Logic has the following behavior:
	•	If the Reset(i3) is True, then the MUX will go to channel 3. The Reset overrides all other commands.
	•	If the Pause(i2) is True, then the MUX will go to channel 0. The Pause overrides all other commands except Reset.
	•	If the register is greater than 179 (i1), then the MUX will go to channel 3.
	•	If the register is equal to 179(i0), then the MUX will go to channel 3.
	•	If all inputs are False, then the circuit runs as a counter.

Find:

	•	Create a truth table that shows the behavior of the Selector Logic. The inputs are i3, i2, i1, i0. The outputs are s3, s2, s1, s0.
	•	Solve the equations for the selector logic.

Implement Project Code (30 points)

Description
This style of coding is a sprint towards the finish line (Agile). The code at this point will run and create an output but is not expected to be 100% complete at the end of Phase 2. The code must be 100% complete by Phase 3. 
Construction
The construction of the code should parallel the testbench and breadboard system done in Phase 1. This time, instead of having a for-loop generating a truth table, the testbench will be a set of commands sent to the bread board working with a clock. Commands against a clock is a program.
What Is Expected
The testbench should include a clock signal, and a series of opcodes, which are sent from the testbench to the breadboard. The breadboard should perform the actions required and return an output. The output would include any current data and a status. The data could be the value of a calculation, the color stored in memory, or a set of error codes. 
The breadboard should be made of components. Instead of gate equations in part 1 of the project, the breadboard should contain the parts listed in the System Design. The input list will be the input parameters of the breadboard module. The outputs will be the output parameters of the breadboard module. The interfaces will be the local variables of the breadboard module. 
The components on the breadboard will be additional modules. If an AND gate is to process two 32-bit binary numbers, the AND gate should be a separate module, instantiated in the breadboard instead of being coded directly into the breadboard. Decoders, Multiplexers, Adders, Subtractors, Multipliers, Dividers, Modulus, the Logic Operations would all be separate modules in the code and instantiated in the breadboard.
Memory elements, the registers, will be made of D Flip-Flops (DFF). The DFF sample code as well as other examples will be available on eLearning in the Project Part 2 folder. The registers are the component that will require the clock signal from the testbench.
The results of the activation of the components should be returned as parameters back to the testbench, and the testbench will output the data to screen.
Structural vs. Behavioral
Sample codes with structural Verilog will be made available to the students for use in constructing components. In these components, there will be no for loops or if statements or other procedural programming used as shortcuts. Specifically, decoders, adders, subtractors, and multiplexers should be structural. The breadboard, by its nature, is structural.
Modules ought to be behavioral would be operations such as modulus and division. The testbench can also be considered to be behavioral. If the Verilog code looks more like C++ and Java then the code is behavioral.
If a new, original component designed by the cohort to do a single action would be larger than the entire program, then it should be made behavioral, and comments included in the module to say why.

Output
Depending on the system being built the output will be quite different. Yet some output will be needed.
Regardless of what design the cohort has chosen, the following should be made visible.
	•	The current opcode is being loaded.
	•	The current input values are being processed.
	•	The current output values that are being returned.
	•	Any error codes or status codes used to safeguard the program.
	•	Any test messages to indicate a part of the program is still incomplete.
Grading
Since every cohort could have a different project, the current code will be compared to what was marked completed in the System Design.

COMPLEXITY

The projects must be as complex as an ALU. An ALU has a 32-bit memory register, a 16-bit feedback loop, and sixteen op-codes representing 16 operations. Math and logic operations are represented by having additional Verilog modules, which also include error codes and status messages to be routed to the output.
